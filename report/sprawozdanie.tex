\section{Wstęp}

Niniejsze sprawozdanie zawiera opis projektu indywidualnego DISHLY - platformy internetowej do usprawniania obsługi zamówień w restauracjach. Projekt jest aplikacją typu SaaS (Software as a Service), która umożliwia restauracjom przyjmowanie zamówień online, a klientom zamawianie jedzenia z dostawą. Raport zawiera szczegółowy opis wymagań, założeń systemu, zakresu pracy, zaimplementowanych funkcjonalności oraz zrzuty ekranu interfejsu użytkownika.

\subsection{Cel Projektu}

Celem projektu DISHLY jest zaprojektowanie i implementacja aplikacji usprawniającej obsługę zamówień w restauracjach. Aplikacja koncentruje się na:
\begin{itemize}
    \item Ułatwieniu zamawiania jedzenia przez stronę internetową
    \item Umożliwieniu klientom wyboru dań z menu oraz złożenia zamówienia
    \item Wdrożeniu modelu biznesowego SaaS, gdzie restauracje opłacają dostęp do platformy
    \item Utworzeniu szczegółowego systemu menu z informacjami o wartościach odżywczych
    \item Implementacji systemu uprawnień dla różnych ról użytkowników
\end{itemize}

\subsection{Zakres Projektu}

Projekt obejmuje:
\begin{enumerate}
    \item \textbf{Zaprojektowanie systemu} - baza danych i logika biznesowa z obsługą wielu ról użytkowników
    \item \textbf{Panel Klienta} - wyszukiwanie restauracji, zamawianie, płatność i śledzenie zamówień
    \item \textbf{Panel Restauracji} - zarządzanie menu, statusami zamówień i powiadomieniami
    \item \textbf{Narzędzia Administracyjne} - weryfikacja restauracji, zarządzanie planami cenowymi
    \item \textbf{Personalizacja UI} - dynamiczne wyświetlanie elementów interfejsu na podstawie uprawnień
\end{enumerate}

\section{Wymagania Projektu}

\subsection{Wymagania Funkcjonalne}

\subsubsection{Logowanie i Rejestracja}

System obsługuje pięć typów użytkowników:
\begin{itemize}
    \item \textbf{Klient (CLIENT)} - zamawiający jedzenie z restauracji
    \item \textbf{Pracownik (WORKER)} - obsługuje zamówienia w konkretnym punkcie
    \item \textbf{Menadżer (MANAGER)} - zarządza danymi konkretnego punktu
    \item \textbf{Właściciel (OWNER)} - zarządza całą restauracją i wszystkimi punktami
    \item \textbf{Administrator (ADMIN)} - administruje całą platformą
\end{itemize}

Każda rola ma dostęp do dedykowanego interfejsu z odpowiednimi funkcjonalizaciami.

\subsubsection{Tworzenie i Zarządzanie Daniami}

\begin{itemize}
    \item Restauracje mogą tworzyć dania z parametrami takimi jak:
    \begin{itemize}
        \item Nazwa i opis
        \item Cena
        \item Czas przygotowania
        \item Kalorie
        \item Ilość białka, węglowodanów i tłuszczy
        \item Poziom ostrości
        \item Zdjęcie
    \end{itemize}
    \item Każde danie pode kategorią (np. Przystawki, Główne, Desery)
    \item Dania wymagają zatwierdzenia administratora
\end{itemize}

\subsubsection{Filtrowanie i Wyszukiwanie}

Klienci mogą filtrować dania według:
\begin{itemize}
    \item Restauracji
    \item Kategorii
    \item Parametrów odżywczych (kalorie, białko, węglowodany, tłuszcze)
    \item Poziomu ostrości
    \item Dostępności w danym punkcie
\end{itemize}

\subsubsection{Model SaaS}

\begin{itemize}
    \item Administrator tworzy pakiety subskrypcyjne z limitami:
    \begin{itemize}
        \item Liczba dostępnych dań
        \item Liczba lokalizacji restauracji
        \item Liczba kont pracowników
    \end{itemize}
    \item Restauracje wybierają pakiet odpowiadający ich potrzebom
    \item Subskrypcje zarządzane są automatycznie (naliczanie opłat)
\end{itemize}

\subsubsection{Statusy Zamówień}

\begin{itemize}
    \item Zamówienie przechodzi przez różne statusy:
    \begin{itemize}
        \item Nowe
        \item W przygotowaniu
        \item Gotowe
        \item W dostawie
        \item Dostarczone
        \item Anulowane
    \end{itemize}
    \item Klient otrzymuje powiadomienia o zmianach statusu
    \item Pracownik może zmienić status zamówienia
\end{itemize}

\subsubsection{System Uprawnień (RBAC)}

\begin{itemize}
    \item Właściciel - pełny dostęp do restauracji i wszystkich punktów
    \item Menadżer - dostęp do statystyk i menu konkretnego punktu
    \item Pracownik - dostęp tylko do obsługi zamówień w konkretnym punkcie
    \item Klient - dostęp do przeglądania menu i składania zamówień
    \item Administrator - dostęp do zarządzania całą platformą
\end{itemize}

\subsection{Wymagania Niefunkcjonalne}

\begin{itemize}
    \item \textbf{Skalowalność} - aplikacja powinna obsługiwać tysiące restauracji i zakupów
    \item \textbf{Bezpieczeństwo} - bezpieczne przechowywanie haseł (bcryptjs), HTTPS
    \item \textbf{Wydajność} - szybkie wyszukiwanie i filtrowanie dań
    \item \textbf{Dostępność} - aplikacja dostępna 24/7
    \item \textbf{Responsywność} - interfejs dostosowany do urządzeń mobilnych
    \item \textbf{Integracja z płatnościami}  -  obsługa Stripe
\end{itemize}

\section{Struktura Bazy Danych}

\subsection{Przegląd Ogólny}

Baza danych DISHLY zbudowana jest na PostgreSQL i zarządzana za pośrednictwem ORM Prisma. System zawiera 25 modeli danych, które reprezentują główne byty systemu:

\begin{itemize}
    \item Zarządzanie Użytkownikami i Autentykacją
    \item Zarządzanie Restauracjami i Lokalizacjami
    \item Katalog Dań i Menu
    \item System Zamówień
    \item Subskypcje i Płatności
    \item Recenzje i Oceny
    \item Zarządzanie Treścią
\end{itemize}

\subsection{Główne Byty i Ich Relacje}

\subsubsection{Użytkownicy (User)}

\textbf{Opis}: Model reprezentujący wszystkich użytkowników systemu z różnymi rolami.

\textbf{Pola}:
\begin{itemize}
    \item \texttt{id} (String, PK) - Identyfikator unikalny (CUID)
    \item \texttt{email} (String, UNIQUE) - Email użytkownika
    \item \texttt{passwordHash} (String) - Zahashowane hasło
    \item \texttt{role} (Role enum) - Rola: ADMIN, OWNER, MANAGER, WORKER, CLIENT
    \item \texttt{firstName, lastName, phone} (String) - Dane personalne
    \item \texttt{isApproved} (Boolean) - Status zatwierdzenia konta
    \item \texttt{isActive} (Boolean) - Status aktywności
    \item \texttt{createdAt, updatedAt} (DateTime) - Sygnatury czasowe
\end{itemize}

\textbf{Relacje}:
\begin{itemize}
    \item \texttt{addresses} → Address (jeden-do-wielu): Adresy dostawy użytkownika
    \item \texttt{ownedRestaurants} → Restaurant (jeden-do-wielu): Restauracje należące do użytkownika
    \item \texttt{orders} → Order (jeden-do-wielu): Zamówienia złożone przez użytkownika
    \item \texttt{reviews} → Review (jeden-do-wielu): Recenzje wystawione przez użytkownika
\end{itemize}

\subsubsection{Restauracje (Restaurant)}

\textbf{Opis}: Model przedstawiający restauracje w systemie.

\textbf{Pola}:
\begin{itemize}
    \item \texttt{id} (String, PK) - Identyfikator unikalny
    \item \texttt{name} (String) - Nazwa restauracji
    \item \texttt{slug} (String, UNIQUE) - Slug dla URL
    \item \texttt{logoUrl, coverImageUrl} (String) - Zdjęcia
    \item \texttt{ownerId} (String, FK) - Właściciel restauracji
    \item \texttt{status} (RestaurantStatus enum) - PENDING, APPROVED, REJECTED
    \item \texttt{isActive} (Boolean) - Aktywność restauracji
    \item \texttt{nip} (String) - Numer identyfikacji podatkowej
\end{itemize}

\textbf{Relacje}:
\begin{itemize}
    \item \texttt{owner} → User: Właściciel restauracji
    \item \texttt{locations} → Location (jeden-do-wielu): Lokalizacje restauracji
    \item \texttt{meals} → Meal (jeden-do-wielu): Menu dań
    \item \texttt{subscriptions} → Subscription (jeden-do-wielu): Subskrypcje restauracji
    \item \texttt{reviews} → Review (jeden-do-wielu): Recenzje restauracji
    \item \texttt{tags} → RestaurantTag (wiele-do-wielu): Tagi (amenities)
    \item \texttt{cuisineTypes} → CuisineType (wiele-do-wielu): Typy kuchni
\end{itemize}

\subsubsection{Lokalizacje (Location)}

\textbf{Opis}: Konkretne punkty obsługi restauracji.

\textbf{Pola}:
\begin{itemize}
    \item \texttt{id} (String, PK) - Identyfikator unikalny
    \item \texttt{restaurantId} (String, FK) - Restauracja
    \item \texttt{name, address, city, postalCode, phone} (String) - Dane lokalizacji
    \item \texttt{latitude, longitude} (Float) - Współrzędne GPS
    \item \texttt{deliveryRadius} (Int) - Promień dostawy w km
    \item \texttt{deliveryFee, minOrderValue} (Decimal) - Parametry finansowe
    \item \texttt{openingHours} (JSON) - Godziny otwarcia
    \item \texttt{isAllDay} (Boolean) - Otwarte 24/7
\end{itemize}

\textbf{Relacje}:
\begin{itemize}
    \item \texttt{restaurant} → Restaurant: Restauracja rodzic
    \item \texttt{orders} → Order (jeden-do-wielu): Zamówienia dla tej lokalizacji
    \item \texttt{mealLocations} → MealLocation (jeden-do-wielu): Dostępne dania
    \item \texttt{workers} → User (jeden-do-wielu): Pracownicy tej lokalizacji
\end{itemize}

\subsubsection{Dania (Meal)}

\textbf{Opis}: Pozycje menu dostępne u restauracji.

\textbf{Pola}:
\begin{itemize}
    \item \texttt{id} (String, PK) - Identyfikator unikalny
    \item \texttt{restaurantId} (String, FK) - Restauracja
    \item \texttt{categoryId} (String, FK) - Kategoria dania
    \item \texttt{name, slug, description, imageUrl} (String) - Dane dania
    \item \texttt{basePrice} (Decimal) - Cena bazowa
    \item \texttt{preparationTime} (Int) - Czas przygotowania w minutach
    \item \texttt{calories, protein, carbs, fat, weight} (Int/Decimal) - Wartości odżywcze
    \item \texttt{spiceLevel} (Int) - Level ostrości (0-5)
    \item \texttt{isVegetarian, isVegan, isGlutenFree} (Boolean) - Atrybuty dietetyczne
    \item \texttt{approvalStatus} (MealApprovalStatus) - PENDING, APPROVED, REJECTED
\end{itemize}

\textbf{Relacje}:
\begin{itemize}
    \item \texttt{restaurant} → Restaurant: Restauracja
    \item \texttt{category} → Category: Kategoria
    \item \texttt{addons} → MealAddon (jeden-do-wielu): Dodatki do dania
    \item \texttt{variants} → MealVariant (jeden-do-wielu): Warianty (rozmiary, etc.)
    \item \texttt{locations} → MealLocation (jeden-do-wielu): Dostępność w lokalizacjach
\end{itemize}

\subsubsection{Zamówienia (Order)}

\textbf{Opis}: Zamówienia złożone przez klientów.

\textbf{Pola}:
\begin{itemize}
    \item \texttt{id} (String, PK) - Identyfikator unikalny
    \item \texttt{orderNumber} (String, UNIQUE) - Numer zamówienia wyświetlany klientowi
    \item \texttt{userId} (String, FK) - Klient (nullable dla gościa)
    \item \texttt{locationId} (String, FK) - Lokalizacja restauracji
    \item \texttt{type} (OrderType enum) - DELIVERY lub PICKUP
    \item \texttt{status} (OrderStatus enum) - PENDING, ACCEPTED, PREPARING, READY, IN\_DELIVERY, DELIVERED, CANCELLED
    \item \texttt{paymentStatus} (PaymentStatus) - PENDING, PAID, FAILED, REFUNDED
    \item \texttt{subtotal, deliveryFee, totalPrice} (Decimal) - Kwoty
    \item \texttt{estimatedDeliveryAt} (DateTime) - Oczekiwany czas dostawy
    \item \texttt{guestName, guestEmail, guestPhone} (String) - Dane gościa (jeśli brak konta)
\end{itemize}

\textbf{Relacje}:
\begin{itemize}
    \item \texttt{user} → User: Klient (nullable)
    \item \texttt{location} → Location: Lokalizacja restauracji
    \item \texttt{items} → OrderItem (jeden-do-wielu): Pozycje zamówienia
    \item \texttt{statusLogs} → OrderStatusLog (jeden-do-wielu): Historia zmian statusu
\end{itemize}

\subsubsection{Subskrypcje (Subscription)}

\textbf{Opis}: Plany subskrypcji dla restauracji (model SaaS).

\textbf{Tabela Plan Subskrypcji (SubscriptionPlan)}:
\begin{itemize}
    \item \texttt{id} (String, PK)
    \item \texttt{name} (String, UNIQUE) - Nazwa planu (np. ``Starter'', ``Professional'')
    \item \texttt{price} (Decimal) - Miesięczna cena
    \item \texttt{interval} (SubscriptionInterval) - MONTHLY / YEARLY
    \item \texttt{maxMeals} (Int) - Limitowana liczba dań
    \item \texttt{maxLocations} (Int) - Limitowana liczba lokalizacji
    \item \texttt{maxStaffAccounts} (Int) - Limitowana liczba kont pracowników
\end{itemize}

\textbf{Tabela Subskrypcji (Subscription)}:
\begin{itemize}
    \item \texttt{restaurantId} (String, FK) - Restauracja
    \item \texttt{planId} (String, FK) - Wybrany plan
    \item \texttt{startDate, endDate} (DateTime) - Okres subskrypcji
    \item \texttt{stripeSubscriptionId} (String) - ID subskrypcji Stripe
    \item \texttt{isActive} (Boolean) - Status aktywizacji
\end{itemize}

\subsubsection{Kategorie (Category)}

\textbf{Opis}: Kategorie dań (Przystawki, Dania główne, Desery, Napoje, etc.)

\textbf{Pola}:
\begin{itemize}
    \item \texttt{id} (String, PK)
    \item \texttt{name, slug} (String)
    \item \texttt{imageUrl} (String) - Zdjęcie kategorii
    \item \texttt{sortOrder} (Int) - Kolejność wyświetlania
\end{itemize}

\subsubsection{Recenzje (Review)}

\textbf{Opis}: Opinie i oceny restauracji od klientów.

\textbf{Pola}:
\begin{itemize}
    \item \texttt{restaurantId, userId} (String, FK) - Restauracja i użytkownik
    \item \texttt{rating} (Int) - Ocena 1-5
    \item \texttt{content} (String) - Tekst opinii
    \item \texttt{isVisible} (Boolean) - Widoczność dla innych
\end{itemize}

\subsection{Constraints i Business Rules}

\begin{itemize}
    \item \textbf{Unikalność maili}: Każdy email musi być unikalny w systemie
    \item \textbf{Unikalność slugów}: Slugi restauracji i kategorii muszą być unikalne
    \item \textbf{Relacja Meal-Location}: Każde danie może być dostępne w wielu lokalizacjach (MealLocation)
    \item \textbf{Kasada na usunięcie}: Usunięcie restauracji kasuje jej lokalizacje, dania i zamówienia
    \item \textbf{Status zamówienia}: Przepływ: PENDING → ACCEPTED → PREPARING → READY → IN\_DELIVERY/DELIVERED lub CANCELLED
    \item \textbf{Subskrypcja}: Restauracja może mieć tylko jedną aktywną subskrypcję naraz
\end{itemize}

    \begin{verbatim}
